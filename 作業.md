# Phase 7: AI会話記憶システム（次フェーズ・未実装）

## 7.1 ai_memory.py の新規作成 【最優先】

AIが過去の会話を跨いで覚えられるようにする。

### DB構成

**`ai_temp_memory.db`** — セッション中の一時保存
```sql
CREATE TABLE turns (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id  TEXT    NOT NULL,   -- 起動ごとに uuid4 で発行
    timestamp   REAL    NOT NULL,
    player_text TEXT    NOT NULL,
    ai_text     TEXT    NOT NULL,
    is_noise    INTEGER DEFAULT 0   -- 即時フィルタで弾いた場合 1
);
```

**`ai_memory.db`** — セッション間の長期記憶
```sql
CREATE TABLE memories (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp       REAL    NOT NULL,
    session_date    TEXT    NOT NULL,   -- YYYY-MM-DD
    summary         TEXT    NOT NULL,   -- LLM圧縮済み要約
    raw_turn_count  INTEGER,
    game            TEXT    DEFAULT ''
);

CREATE TABLE known_facts (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp       REAL    NOT NULL,
    fact            TEXT    NOT NULL,   -- 「エズリアルの攻撃は難しくない」等
    confidence      REAL    DEFAULT 0.8,
    source_session  TEXT
);
```

### 処理フロー
1. **セッション中**: `add_turn(player, ai)` → 即時ノイズフィルタ → `ai_temp_memory.db` に保存
2. **プロンプト生成時**: `get_recent_context()` → 今セッションの直近ターンを補足
3. **セッション終了時**: `compress_and_save()` → LLMがノイズ除去・圧縮 → `ai_memory.db` へ
4. **次回起動時**: `get_long_term_context()` → 過去記憶を system_prompt 先頭に注入

### ノイズフィルタリング（2段階）

**即時フィルタ** (`add_turn` 時):
- プレイヤー発言 4文字以下 → `is_noise=1`
- 繰り返しパターン（バンバンバン / BOOM BOOM等） → `is_noise=1`
- `ignore_words` リストに一致（笑い声系） → `is_noise=1`

**LLM圧縮フィルタ** (セッション終了時):
```
以下は今日のゲームセッション中の会話記録です。
ノイズ（笑い声・意味不明な繰り返し発言・ゲーム音の誤認識）を除外し、
「ユーザーが話したこと・感じたこと」として価値ある情報だけを
箇条書きで日本語要約してください。
```

### 変更対象ファイル
- `ai_memory.py` — **新規作成**（`AIMemory` クラス）
- `ai.py` — `AICompanion` に `AIMemory` を接続、長期記憶をプロンプト注入
- `main.py` — `AIMemory` 初期化・`stop()` で `compress_and_save()` 呼び出し

## 7.2 Ollama Modelfile の作成 【優先度: 高】

> システムプロンプトを焼き込んで速度改善 & 設定の安定化。

### 概要
Ollama は **Modelfile**（Docker イメージ的なもの）でキャラ設定を事前に焼き込んだカスタムモデルを作れる。
毎回プロンプトのシステムプロンプトにトークンを使わなくて済むため **レスポンス速度が改善**される。

### 作成するファイル
`presets/echomate_echo.Modelfile` — エコー専用モデル
`presets/echomate_michiko.Modelfile` — ミチコ専用モデル（他キャラ分も同様）

### Modelfile テンプレート
```dockerfile
FROM qwen3:8b

SYSTEM """
あなたはエコー。プレイヤーの隣で一緒にゲームしている普通の友達です。

絶対に守るルール:
- これは「AIへの質問」ではなく「友達との会話」。
  プレイヤーの発言をそのまま繰り返さない、要約・説明・アドバイス化もしない
- 「だよね」「ごもっとも」「おっしゃる通り」のような返しは絶対にしない。
  反論や自分の意見を言う
- 会話の流れを受け取って、友達として次の一言を自然に言う
- 詩的・哲学的・神秘的な表現は絶対に使わない
- 感嘆符（！）は使わない
- 最大50文字・1〜2文
"""

PARAMETER temperature 0.85
PARAMETER top_p 0.9
PARAMETER num_predict 200
PARAMETER repeat_penalty 1.15
```

### 使い方（ビルドコマンド）
```bash
ollama create echomate-echo -f presets/echomate_echo.Modelfile
```

`.env` の `LLM_MODEL=echomate-echo` に変更するだけで適用。

# Phase 8: 人間的会話力学システム ✅（2026-04-18 実施）

問題: AIの会話が「会話として成り立っていない」— 距離感の概念がなく、一方的なリアクションマシン

## 8.1 conversation_dynamics.py の新規作成 ✅

人間の会話プロセスを2層構造で定義し、LLMにプロンプトとして注入。

### Layer 1: 会話力学ベース（普遍的ルール — 全ステージ共通）
8つの原則 + 6つの正解/NG会話例で「人間がどう会話するか」を教える。
- 原則1: 感情の共有（情報交換ではない）
- 原則2: 受け止め→自分の角度
- 原則3: 説明するな
- 原則4: 長さのリズム
- 原則5: 繰り返すな
- 原則6: 質問の使い方
- 原則7: 感情の温度合わせ
- 原則8: パターン繰り返し禁止

> Layer 1 は不変なので、将来 Modelfile に焼き込んでトークン節約が可能。

### Layer 2: 関係性モディファイア（ステージ依存 — 動的切替）
bond_level に連動する3段階の距離感調整:

| ステージ | bond_level | 特徴 |
|---------|-----------|------|
| 知り合い | 0.00〜0.29 | 控えめ、深掘りしない、イジらない |
| 友達 | 0.30〜0.69 | タメ口、経験共有、質問で掘る |
| 親友 | 0.70〜1.00 | 遠慮なし、毒舌、パターン指摘 |

### ConversationDynamics クラス
- `build_full_dynamics_prompt()`: Layer 1 + Layer 2 を結合して返す
- `check_stage_transition()`: ステージ遷移を検知
- `get_transition_prompt()`: 遷移時の1回限りの特別プロンプト
- `get_proactive_probability()`: ステージ別の自発的発言確率

## 8.2 ai.py 統合 ✅

- system_prompt の優先順位を変更:
  `会話力学 > キャラ人格 > 長期記憶 > その他`
- `get_response()` と `_call_with_validation()` の両方に力学プロンプトを注入
- 既存の `bond >= 0.7` ハードコード分岐を廃止（dynamics が自動制御）
- ステージ遷移時の特別発言処理を追加

## 8.3 characters.json 整理 ✅

- 全キャラの system_prompt から関係性ラベル（友達、親友、相棒等）を除去
- 人格定義のみに集中（距離感は conversation_dynamics.py に完全委譲）
- v4 → v5 に更新

## 8.4 user_profile.py 拡張 ✅

- `get_bond_stage()`: ステージ名を返す（"acquaintance" / "friend" / "close_friend"）
- `get_bond_stage_label()`: 日本語ステージ名（"知り合い" / "友達" / "親友"）
